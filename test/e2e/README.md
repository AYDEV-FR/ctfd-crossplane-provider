# End-to-end tests

`run.sh` exercises the whole provider against a real CTFd instance running in a
local [kind](https://kind.sigs.k8s.io/) cluster. Everything after the cluster is
**fully declarative** — there is no manual setup step:

1. create a kind cluster;
2. build the provider **and bootstrap** images from source and load them;
3. apply the provider CRDs and (optionally) install **Crossplane** core;
4. `kubectl apply` the whole stack at once:
   - a minimal **CTFd** (SQLite, single worker),
   - the **bootstrap `Job`** — runs the CTFd `/setup` wizard (or logs in if
     already configured), mints an admin API token and writes it into the
     `ctfd-creds` Secret via the Kubernetes API,
   - the static `ProviderConfig`/`ClusterProviderConfig` (referencing that
     Secret),
   - the **provider** Deployment;
5. wait for the bootstrap Job to complete and the provider to roll out;
6. apply the example `Challenge` (with inline flags/hints), `Page` and `Theme`
   resources and wait for them to become `Ready`;
7. assert — through the CTFd API, using `ctfdctl -mode verify` — that the
   challenges, their flags and hints (including the hint **prerequisite
   chain**), the page and the theme settings were actually created;
8. assert that an **unset Challenge `state` is not reconciled**: apply a
   challenge that omits `state`, flip its state in CTFd out-of-band
   (`ctfdctl -mode setstate`), nudge a reconcile and confirm the provider does
   not revert it (`ctfdctl -mode checkstate`).

The credentials never leave the cluster: the token is generated by the in-cluster
Job and written straight into the Secret, so bringing the stack up requires no
manual wizard, no token copy-paste and no `kubectl create secret`.

## Run

```shell
make test.e2e
# or directly:
./test/e2e/run.sh
```

Requirements: a running **docker** daemon, **kind**, **kubectl**, **go**, and
**helm** (unless `INSTALL_CROSSPLANE=0`).

## Overrides

| Variable             | Default                | Effect                                   |
|----------------------|------------------------|------------------------------------------|
| `KEEP`               | `0`                    | `1` keeps the kind cluster on exit.      |
| `INSTALL_CROSSPLANE` | `1`                    | `0` skips installing Crossplane core.    |
| `CTFD_IMAGE`         | `ctfd/ctfd:latest`      | CTFd image to deploy.                     |
| `CLUSTER`            | `provider-ctfd-e2e`    | kind cluster name.                        |
| `PF_PORT`            | `8000`                 | local port used to reach CTFd.            |

On failure the script prints the provider logs and the managed-resource status
before tearing the cluster down (use `KEEP=1` to investigate).

## Layout

```
test/e2e/
  run.sh                  # orchestrator (build/load images, apply, wait, verify)
  ctfdctl/                # go helper: bootstrap (setup + write Secret) / verify
  manifests/
    ctfd.yaml             # CTFd Deployment + Service
    bootstrap.yaml        # SA + RBAC + Job that sets up CTFd and writes the Secret
    providerconfig.yaml   # static ProviderConfig + ClusterProviderConfig
    provider.yaml         # provider RBAC + Deployment (image injected by run.sh)
    challenge-no-state.yaml # Challenge omitting `state` (unmanaged-state assertion)
```

The `bootstrap.yaml` Job + `ctfdctl -mode bootstrap` are reusable outside the
e2e: point them at any CTFd Service and they will configure it and produce the
provider credentials Secret declaratively.

> The provider is deployed here as a plain controller `Deployment` built from the
> local image, which keeps the test self-contained. In production you would
> instead install it as a Crossplane package (a `Provider` object referencing a
> published xpkg image).
